# 校园小吃点评平台 - 完整规划方案

> 核心定位：纯粹客观、零商业化干扰的大学生校园美食评价平台
> 初期目标：建立真实口碑生态，不以盈利回本为目的

---

## 一、平台定位与核心需求分析

### 1.1 目标用户画像

核心用户为在校大学生，年龄18-25岁，具有以下特征：

- 消费能力有限，对性价比高度敏感
- 社交活跃，信任同龄人口碑推荐
- 用餐场景集中在食堂、校门口小吃街、外卖
- 对"虚假好评""刷单"有天然反感
- 移动端使用为主，碎片化浏览

### 1.2 核心痛点

| 痛点 | 现有方案的不足 |
|------|---------------|
| 不知道吃什么 | 大众点评/美团覆盖校园周边不精准，商业化严重 |
| 评分不可信 | 商家刷好评、删差评普遍，权重不透明 |
| 缺少学生视角 | 现有平台以商家入驻为主，学生声音被稀释 |
| 信息分散 | 校园美食信息散落在微信群、朋友圈、小红书 |

### 1.3 平台定位声明

**我们不是什么：**
- 不是商家入驻平台（拒绝商家付费买流量）
- 不是外卖平台（不做交易撮合）
- 不是广告平台（拒绝任何形式的商业权重干预）

**我们是什么：**
- 纯学生驱动的美食口碑社区
- 评分体系完全透明、可审计
- 以"真实"为唯一核心竞争力

## 二、基础功能与核心交互模块

### 2.1 用户账户与个性化推荐体系

#### 2.1.1 注册与认证

- 学信网学籍认证（详见第三章反作弊机制）
- 手机号绑定（实名制要求）
- 校园邮箱验证（.edu.cn后缀）
- 头像、昵称、所在校区设置
- 可选：饮食偏好标签（辣/不辣、素食、清真等）

#### 2.1.2 用户成长等级

| 等级 | 名称 | 条件 | 基础权重系数 |
|------|------|------|-------------|
| Lv1 | 新芽 | 完成注册+学籍认证 | 0.5 |
| Lv2 | 食客 | 发布5条有效评价 | 0.7 |
| Lv3 | 美食家 | 发布20条有效评价 + 10个点赞 | 0.85 |
| Lv4 | 品鉴官 | 发布50条有效评价 + 50个点赞 + 无违规记录 | 1.0 |
| Lv5 | 校园美食达人 | 发布100条有效评价 + 社区贡献突出 | 1.2 |

升级条件透明公开，任何人可查看等级规则和当前进度。

#### 2.1.3 个性化推荐

- 基于用户饮食偏好标签推荐
- 基于浏览历史和收藏行为的协同过滤
- 基于同校区、同口味偏好用户的热门收藏
- 新用户冷启动：展示校区周边综合评分TOP20
- 推荐算法完全不考虑商家是否付费（零商业化）

### 2.2 校园餐厅与周边小吃分类、检索与详情页

#### 2.2.1 商户分类体系

一级分类：
- 学校食堂（按档口细分）
- 校门口小吃街
- 校园周边餐馆
- 外卖专营店
- 饮品甜品店
- 夜市摊位

二级分类（按菜系/品类）：
- 中式快餐、面食、麻辣烫、烧烤、炸鸡、奶茶咖啡、甜品等

#### 2.2.2 检索功能

- 关键词搜索（店名、菜品名、地址）
- 按距离排序（基于校区定位）
- 按评分排序
- 按价格区间筛选（人均消费：0-10元/10-20元/20-30元/30元以上）
- 按营业状态筛选（营业中/已打烊）
- 按标签筛选（"分量大""出餐快""适合聚餐""适合自习后吃"等学生标签）

#### 2.2.3 商户详情页

页面结构：
1. **头部区域**：店名、地址、营业时间、人均价格、综合评分
2. **评分雷达图**：口味、卫生、服务、性价比、分量 五个维度
3. **热门菜品**：按评价提及频率自动聚合的菜品列表
4. **评价列表**：支持按最新/最热/带图筛选
5. **附近推荐**：同类商户3家推荐

商户数据来源：
- 初期由志愿者众包录入（详见第六章）
- 后续由用户纠错和补充
- 拒绝商家自行入驻和编辑（防止自我美化）

### 2.3 图文/视频评价及评分打分系统

#### 2.3.1 评分维度

每条评价包含五个独立维度打分（1-5分，0.5分步进）：

| 维度 | 说明 | 权重占比 |
|------|------|---------|
| 口味 | 食物本身好不好吃 | 35% |
| 卫生 | 环境整洁度、食品安全感 | 20% |
| 服务 | 态度、效率、出餐速度 | 15% |
| 性价比 | 价格是否合理 | 20% |
| 分量 | 份量是否足够 | 10% |

综合评分 = 各维度加权平均 × 用户权重系数（见第四章）

#### 2.3.2 评价内容要求

最低标准（否则不计入有效评价）：
- 文字不少于20字
- 至少打满3个维度的分
- 建议附带图片或视频（非强制但影响权重）

鼓励的高质量评价：
- 包含具体菜品名称和价格
- 包含实拍图片（至少1张）
- 包含视频（15秒以上）
- 包含用餐时间、排队时长等实用信息
- 包含对比评价（"比XX家好吃"）

#### 2.3.3 评价展示规则

- 默认按"综合热度"排序（综合点赞数、评价质量、时效性）
- 支持按最新、评分最高、最低排序
- 带图/视频评价优先展示
- 高权重用户的评价优先展示
- 已认证用户的评价标注"已认证学生"

### 2.4 社区互动功能

#### 2.4.1 点赞（觉得有用）

- 每个用户对同一评价只能点赞一次
- 点赞数影响评价的展示排序和权重
- 点赞用户的等级也影响点赞的权重贡献

#### 2.4.2 回复

- 支持对评价进行回复（楼中楼形式）
- 商家可通过官方认证账号回复（需提交营业执照验证）
- 商家回复不计入评分，仅作为信息补充
- 用户回复支持@其他用户

#### 2.4.3 收藏

- 支持收藏商户和单条评价
- 收藏夹支持自定义分组（"下次必吃""聚餐推荐""避雷"等）
- 收藏数据用于个性化推荐

#### 2.4.4 分享

- 生成带评分摘要的分享卡片
- 支持分享到微信、QQ、微博
- 分享链接可直接打开小程序/H5页面
- 分享不包含任何推广追踪参数（保持纯粹）

## 三、防刷评与反作弊风控机制

### 3.1 注册门槛控制

#### 3.1.1 学信网学籍认证

认证流程：
1. 用户填写姓名、身份证号、学校名称
2. 平台调用学信网API进行在读学籍验证
3. 验证通过后标记为"已认证学生"
4. 认证状态每学期自动复核一次

成本控制：
- 学信网查询接口约0.5-2元/次
- 初期可由平台承担认证费用
- 用户量大后可要求用户自付或通过校园邮箱二次验证降低认证频次

#### 3.1.2 备选认证方案（降低初期成本）

- 校园邮箱验证（.edu.cn后缀，免费但可信度略低）
- 学生证照片人工审核（成本高但可作为补充）
- 校园WiFi定位验证（连接校园网时可辅助判断）

建议初期采用"校园邮箱验证 + 学信网抽查"的混合方案。

### 3.2 账号行为频率限制

#### 3.2.1 评价频率限制

| 限制项 | 规则 |
|--------|------|
| 单日评价上限 | 5条（新用户前7天为3条） |
| 同一商户重复评价间隔 | 7天 |
| 新注册用户首评等待期 | 注册后24小时 |
| 单次评价最短提交时间 | 填写时间不低于60秒（防机器人快速提交） |

#### 3.2.2 设备指纹追踪

采集以下信息生成设备指纹：
- 浏览器User-Agent
- 屏幕分辨率
- 时区和语言
- Canvas指纹
- WebGL指纹
- IP地址（含运营商信息）
- 触控特征（移动端）

风控规则：
- 同一设备24小时内注册账号数限制：2个
- 同一设备关联账号超过5个触发人工审核
- 同一IP段短时间内大量评价触发告警
- 设备指纹与账号绑定，换号不换设备可追踪

### 3.3 AI智能审查机制

#### 3.3.1 内容质量过滤

自动识别并拦截：
- 纯表情/无意义字符评价
- 复制粘贴的模板化评价（与历史评价相似度>80%）
- 包含明显广告/引流内容
- 包含辱骂、歧视等违规内容
- AI生成的虚假评价（通过文本特征分析识别）

#### 3.3.2 刷分行为识别

异常模式检测：
- 短时间内同一商户评分集中升高（如1小时内10条5星评价）
- 评价内容雷同但来自不同账号
- 新注册账号集中给同一商户好评
- 评分分布异常（正常商户评分应近似正态分布，全5星或全1星为异常）
- 评价时间分布异常（集中在非用餐时间的批量评价）

#### 3.3.3 处理机制

| 风险等级 | 触发条件 | 处理方式 |
|---------|---------|---------|
| 低风险 | 单条内容质量低 | 自动折叠，不计入评分 |
| 中风险 | 疑似刷分行为 | 标记待人工审核，暂停计入评分 |
| 高风险 | 确认刷分/水军 | 删除评价，账号降权或封禁 |
| 极高风险 | 有组织刷单 | 封禁所有关联账号，公示处理结果 |

#### 3.3.4 举报机制

- 每条评价右下角提供"举报"按钮
- 举报类型：虚假评价、广告内容、恶意差评、抄袭、其他
- 举报后进入人工审核队列
- 有效举报奖励：举报者获得社区贡献值
- 恶意举报（频繁误报）会被降权举报权重

## 四、点评权重与信誉系统

### 4.1 权重计算公式

单条评价的有效评分 = 基础维度加权分 × 用户权重系数 × 内容丰富度系数

商户综合评分 = Σ(每条有效评价的有效评分 × 时效衰减因子) / Σ(每条有效评价的权重系数 × 时效衰减因子)

### 4.2 用户权重系数

基础权重由以下因素决定：

| 因素 | 计算方式 | 影响范围 |
|------|---------|---------|
| 用户等级 | Lv1=0.5, Lv2=0.7, Lv3=0.85, Lv4=1.0, Lv5=1.2 | 基础值 |
| 历史评价质量 | 被点赞率、被收藏率、无违规记录 | ±20%调整 |
| 活跃度 | 近30天评价数量（适度活跃>沉默>刷屏） | ±10%调整 |
| 认证状态 | 学信网认证=1.0，邮箱认证=0.8，未认证=0.5 | 乘法因子 |

**关键原则：权重系数完全由用户行为决定，与任何商业因素无关。**

权重计算规则公开可查，任何人可在"关于我们"页面查看完整的权重计算逻辑。

### 4.3 内容丰富度系数

| 内容类型 | 系数 |
|---------|------|
| 纯文字（20-50字） | 0.8 |
| 纯文字（50字以上） | 1.0 |
| 文字 + 1-3张图片 | 1.1 |
| 文字 + 4张以上图片 | 1.2 |
| 文字 + 视频 | 1.3 |
| 文字 + 图片 + 视频 | 1.4 |

### 4.4 时效衰减因子

- 发布后0-7天：权重1.0
- 发布后7-30天：权重0.9
- 发布后30-90天：权重0.7
- 发布后90-180天：权重0.5
- 发布后180天以上：权重0.3

目的：确保商户评分反映近期真实状态，避免历史好评掩盖近期质量问题。

### 4.5 "觉得有用"点赞的权重影响

- 单条评价被点赞数达到10：该评价权重额外+10%
- 单条评价被点赞数达到50：该评价权重额外+20%
- 单条评价被点赞数达到100：该评价权重额外+30%，且标记为"精华评价"
- 点赞用户的等级也影响点赞权重：Lv5用户的点赞权重是Lv1用户的2倍

### 4.6 权重透明度保障

- 每个商户详情页底部显示"评分说明"链接
- 点击可查看该商户的评价数量、评分分布、权重计算概要
- 用户个人主页可查看自己的权重系数和等级进度
- 平台定期发布《评分透明度报告》，公示异常处理情况

## 五、技术架构与全栈开发方案

### 5.1 技术选型

#### 前端

| 技术 | 选择 | 理由 |
|------|------|------|
| 框架 | React 18 + Next.js 14 | SSR/SSG支持，SEO友好，首屏加载快 |
| UI库 | Tailwind CSS + Radix UI | 轻量、可定制、无障碍支持好 |
| 状态管理 | Zustand | 轻量，适合中小型应用 |
| 移动端 | 响应式Web + PWA | 初期不做原生App，降低开发成本 |
| 小程序 | Taro 3（可选） | 如需微信小程序，可复用React代码 |

#### 后端

| 技术 | 选择 | 理由 |
|------|------|------|
| 语言 | Node.js (TypeScript) | 全栈统一语言，开发效率高 |
| 框架 | Nest.js | 企业级框架，模块化设计，适合团队协作 |
| 数据库 | PostgreSQL | 事务支持好，JSON字段支持，免费 |
| 缓存 | Redis | 热门商户评分缓存、会话管理、排行榜 |
| 搜索 | Meilisearch | 轻量级全文搜索，部署简单，性能好 |
| 文件存储 | 阿里云OSS / MinIO | 图片视频存储 |
| 消息队列 | BullMQ (Redis) | 异步任务（图片压缩、AI审核、通知推送） |

#### 基础设施

| 组件 | 选择 | 理由 |
|------|------|------|
| 部署 | 阿里云ECS / 腾讯云 | 国内访问快，学生优惠 |
| CDN | 阿里云CDN | 图片视频加速 |
| 容器化 | Docker + Docker Compose | 简化部署和环境一致性 |
| CI/CD | GitHub Actions | 免费，与代码仓库集成 |
| 监控 | Prometheus + Grafana | 免费开源，功能完善 |
| 日志 | ELK Stack 或 Loki | 错误追踪和分析 |

### 5.2 系统架构图

```
用户浏览器/手机
    │
    ▼
[CDN] ← 静态资源、图片、视频
    │
    ▼
[Nginx反向代理] ← 负载均衡、SSL终止、限流
    │
    ├──→ [Next.js SSR] ← 页面渲染、SEO
    │
    ├──→ [Nest.js API Gateway] ← 认证、限流、路由
    │        │
    │        ├──→ [用户服务] ← 注册、认证、等级
    │        ├──→ [商户服务] ← 商户CRUD、分类、搜索
    │        ├──→ [评价服务] ← 评价CRUD、评分计算
    │        ├──→ [社区服务] ← 点赞、回复、收藏、分享
    │        ├──→ [风控服务] ← 反作弊、AI审核
    │        └──→ [推荐服务] ← 个性化推荐
    │
    ├──→ [PostgreSQL] ← 主数据库
    ├──→ [Redis] ← 缓存、会话、排行榜
    ├──→ [Meilisearch] ← 全文搜索
    └──→ [OSS/MinIO] ← 文件存储
```

### 5.3 高并发方案

用餐高峰期（11:00-13:00, 17:00-19:00）的应对策略：

1. **评分缓存**：商户综合评分缓存在Redis中，每5分钟异步更新，避免每次请求都实时计算
2. **读写分离**：PostgreSQL主从复制，读请求走从库
3. **CDN加速**：图片、视频、静态资源全部走CDN
4. **接口限流**：单用户每分钟最多60次API请求
5. **数据库连接池**：合理配置连接池大小，避免连接耗尽
6. **异步处理**：图片压缩、AI审核、通知推送等非实时操作走消息队列
7. **预计算**：排行榜、热门商户等定时预计算

预估承载能力（单台4核8G ECS）：
- 日活用户：5000-10000
- 峰值QPS：500-1000
- 数据库连接数：200

### 5.4 开发成本估算

| 项目 | 预估成本（月） |
|------|---------------|
| 云服务器（2台ECS） | 300-500元 |
| 数据库（RDS或自建） | 100-200元 |
| CDN + 带宽 | 100-300元 |
| OSS存储 | 50-100元 |
| 域名 + SSL | 100元/年 |
| 学信网API（按量） | 500-2000元（视用户增长） |
| **合计** | **约1000-3000元/月** |

初期可通过校园服务器资源、云厂商学生优惠进一步降低成本。

## 六、冷启动策略

### 6.1 数据来源：用户众包

核心原则：拒绝商家付费入驻，所有商户数据由学生志愿者众包采集。

#### 6.1.1 志愿者招募

- 在目标校园招募10-20名"创始美食侦探"
- 招募渠道：校园论坛、美食社团、朋友圈
- 激励机制：创始成员专属勋章、优先体验权、平台周边礼品
- 要求：每人负责采集5-10家商户的基础信息

#### 6.1.2 采集内容

每个商户的基础数据模板：
- 店名、地址、营业时间
- 主营品类、人均价格
- 3-5张实拍照片
- 推荐菜品（2-3道）
- 初始评分（基于志愿者实际体验）

#### 6.1.3 采集工具

开发一个简易的"商户录入"H5页面：
- 拍照上传
- 填写基本信息
- GPS自动定位
- 提交后进入审核队列

### 6.2 初期内容填充

#### 6.2.1 种子评价

- 志愿者在采集商户信息的同时，撰写首批真实评价
- 每家商户至少3条不同志愿者的评价
- 确保评价真实、有细节、有图片

#### 6.2.2 食堂档口专项

- 食堂档口是学生最高频的用餐场景
- 按档口逐一录入，每个档口至少2条评价
- 重点覆盖：早餐档口、午餐热门档口、夜宵档口

### 6.3 冷启动阶段目标

| 阶段 | 时间 | 目标 |
|------|------|------|
| 第1周 | 准备期 | 完成技术开发、志愿者招募 |
| 第2-3周 | 数据采集期 | 录入100+商户、300+条种子评价 |
| 第4周 | 内测期 | 邀请50名同学内测、收集反馈 |
| 第5-6周 | 公开期 | 校园内公开上线、首批推广 |

## 七、校园本地化运营与推广方案

### 7.1 社团合作

#### 7.1.1 美食社团

- 与校园美食社团建立官方合作关系
- 社团成员成为"认证美食侦探"
- 联合举办"校园美食评选"活动
- 社团在平台开设专属专栏

#### 7.1.2 学生会/社团联

- 通过学生会官方渠道推广
- 纳入新生入学指南推荐应用
- 与校园活动联动（如美食节、社团招新）

### 7.2 线下地推

#### 7.2.1 食堂驻点

- 午餐/晚餐高峰期在食堂门口设展台
- 现场扫码注册、填写评价送小礼品
- 礼品选择：定制书签、贴纸、手机支架（成本低、实用）

#### 7.2.2 宿舍楼推广

- 在宿舍楼公告栏张贴海报
- 利用楼层群/宿舍群传播
- 设计学生风格的宣传物料（避免商业化风格）

### 7.3 试吃体验官招募

#### 7.3.1 活动形式

- 每月招募10-20名"试吃体验官"
- 体验官免费品尝指定商户（平台不付费，由商户提供免费试吃——注意：这是商户让利给学生，不是平台收费）
- 体验官撰写详细评价，必须包含真实体验
- 评价必须标注"试吃体验"标签（透明度要求）

#### 7.3.2 防止变质为商业化

- 试吃体验官的评价不享受额外权重加成
- 试吃评价必须同时包含优点和不足
- 平台保留删除"全好评"试吃评价的权利
- 试吃活动频率限制：每商户每季度最多1次

### 7.4 内容营销

#### 7.4.1 校园美食榜单

- 每周发布"本周校园美食TOP5"
- 每月发布"月度最佳新店"
- 每学期发布"校园美食白皮书"
- 所有榜单基于平台真实数据生成，不接受赞助

#### 7.4.2 社交媒体运营

- 微信公众号/视频号：美食图文、探店视频
- 小红书：精美图片+短评
- B站：校园美食Vlog
- 内容风格：真实、有趣、学生视角

### 7.5 口碑裂变机制

- 邀请好友注册双方各得积分（积分兑换周边礼品，非商业利益）
- 优质评价被点赞超过50次，评价者获得"金舌头"徽章
- 每月评选"最佳评价"，获奖者获得平台定制周边
- 分享评价到朋友圈可参与抽奖（奖品为平台周边，非商家优惠券）

## 八、法律合规事项

### 8.1 用户隐私数据保护

#### 8.1.1 数据收集最小化原则

- 仅收集提供服务所必需的数据
- 学信网认证信息仅用于验证身份，不存储原始身份证号
- 设备指纹仅用于风控，不用于广告追踪
- 位置信息仅用于距离计算，不进行持续追踪

#### 8.1.2 隐私政策要点

- 明确告知用户收集哪些数据、用途、存储期限
- 用户有权查看、修改、删除自己的数据
- 用户注销账号后，个人数据在30天内删除（法律要求保留的除外）
- 不向第三方出售用户数据
- 不使用用户数据进行广告定向

#### 8.1.3 合规要求

- 遵守《个人信息保护法》《数据安全法》
- 上线前完成个人信息保护影响评估
- 指定个人信息保护负责人
- 建立数据安全事件应急预案
- 如涉及跨境数据传输，需完成安全评估

### 8.2 UGC内容合规

#### 8.2.1 内容审核机制

三层审核体系：
1. **AI自动审核**：接入阿里云/腾讯云内容安全API，自动识别违规内容
2. **人工审核**：AI标记为"待审核"的内容由人工复核
3. **用户举报**：社区用户举报，人工处理

#### 8.2.2 违规内容类型

- 涉政敏感内容
- 色情低俗内容
- 暴力血腥内容
- 侮辱诽谤内容
- 虚假信息
- 广告引流内容
- 侵犯他人隐私的内容（如未经同意拍摄商家工作人员）

#### 8.2.3 处理流程

1. 违规内容发布后，AI审核在5分钟内完成初步判断
2. 疑似违规内容自动隐藏，等待人工复核
3. 确认违规：删除内容，通知用户，记录违规
4. 多次违规：账号禁言、降权直至封禁
5. 严重违规（涉政、违法）：立即删除，上报主管部门

#### 8.2.4 商家权益保护

- 商家可对不实评价提出申诉
- 申诉需提供证据（如消费记录、监控截图）
- 平台在7个工作日内完成审核
- 确认不实的评价将被删除或标注
- 商家不可要求删除真实的差评

### 8.3 其他合规事项

#### 8.3.1 ICP备案

- 网站上线前完成ICP备案
- 如涉及经营性业务，需办理ICP经营许可证
- 初期作为非经营性网站备案即可

#### 8.3.2 网络安全等级保护

- 按照《网络安全法》要求完成等保定级备案
- 初期建议定为二级（涉及用户个人信息）
- 配合公安机关进行安全检查

#### 8.3.3 知识产权

- 用户发布的评价内容，用户保留著作权
- 平台获得在平台内展示、推荐的授权
- 平台Logo、名称需注册商标保护
- 注意避免侵犯商户商标权（如实展示店名属于合理使用）

#### 8.3.4 食品安全相关

- 平台不对商户的食品安全负责（信息平台免责）
- 但应在用户协议中明确：平台信息仅供参考
- 如收到食品安全投诉，应及时下架相关商户信息并通知监管部门
